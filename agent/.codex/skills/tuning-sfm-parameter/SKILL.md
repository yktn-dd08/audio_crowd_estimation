# SFMパラメータ調整

Pythonで実装されたSocial Force Model（SFM）の人流シミュレーションに対して、パラメータ探索、動画比較、軌跡評価を行い、SFMの最適なパラメータを探索する。

## 禁止事項
- ./workspace以外のディレクトリにファイル書き込みをすること。
- ../agent以外のフォルダに格納されたファイルを修正すること。

## 前提

- ユーザーは、指定したパラメータでSFMシミュレーションを実行し、動画または軌跡CSV/GeoDataFrameを出力できるPythonプログラムを持っている。
- 主な調整対象は次の4つとする。
  - `c_obs`: 人同士の反発力の強さ。
  - `r_obs`: 人同士の反発力が効く距離、または減衰長。
  - `c_wall`: 壁・障害物からの反発力の強さ。
  - `r_wall`: 壁・障害物からの反発力が効く距離、または減衰長。
- ユーザーの実装では `dt=1` を使うことがある。これは数値安定性に対する強い制約として扱う。
- `dt=1` の場合、力を強くしすぎず、反発の有効距離を広げすぎない。必要に応じて速度減衰、加速度クリップ、有効領域への投影を前提にする。
- シミュレーションの出力ファイルは./workspace以下に保存する。この際に、入力したJSONファイルから発行されたハッシュID(SHA256)をフォルダ名として、中にCSVファイル、動画ファイルを保存する。具体的には./workspace/(SHA256)というフォルダを作成し、CSVファイルや動画ファイルを保存する。入力となったJSONデータもそのフォルダ内に保存する。
- 繰り返しシミュレーションを行う過程で、上記のハッシュIDとパラメータの対応を一つのファイルに記録しておく。これにより、後で動画や軌跡を見返すときに、どのパラメータがどの結果を出したのかを簡単に参照できるようにする。またその時の動画の評価結果も保存する。

## SFMのシミュレーションの実行方法
- プロジェクトルート(audio_crowd_estimation)にディレクトリを移動する。。
- ./.venv/bin/activate を実行して仮想環境を有効にする。
- ./simulation/crowd_trajectory.pyを実行してSFMのシミュレーションを実行する。
  - 実行方法は```python -m simulation.crowd_trajectory -c <jsonファイルパス>```のようにする。
- 入力とするのはjsonファイルであり、次のような構造を持つ。なお動画を出力するため、output_mp4は省略しない。

```json
{
    "option": "social_force",
    "param": {
        "person_num": 500,
        "simulation_time": 360,
        "desired_speed": 1.3,
        "roi_shp": "./data/gis/sfm/roi/roi1_sfm.shp",
        "wall_shp": "./data/gis/sfm/wall/wall1_sfm.shp",
        "goal_flag": true
    },
    "task_list": {
        "sfm_crowd": {
            "output_csv": "./agent/workspace/manual/template/roi1_sfm_crowd.csv",
            "output_mp4": "./agent/workspace/manual/template/roi1_sfm_crowd.mp4",
            "start_time": "2024-01-01 00:00:00",
            "c_obs": 300.0, "r_obs": 0.3, "c_wall": 500.0, "r_wall": 0.7
        }
    }
}
```


## 基本ワークフロー
1. ユーザ入力のプロンプトと入力されたJSONを修正してシミュレーションを実行する。JSONが入力されなかった場合は、scripts/sfm_wall_crowd_trajectory.jsonを修正して、シミュレーションを実行する。
2. 比較中はSFMのパラメータ以外を固定する。
3. 各実行結果ごとにフォルダに保存し、メタデータJSONも残す。
4. 動画の目視確認と軌跡メトリクスの両方で比較する。
5. 評価結果が良くなるようにパラメータを変更し、繰り返す
6. 繰り返し回数は最大1000回までとする。
7. 最適なパラメータセットを見つけたら、動画と軌跡の両方で評価し、結果を要約する。また上位の評価結果から各SFMパラメータの良い範囲を推定し、同じく結果を要約する。

## 動画評価チェックリスト

各動画を1から5で採点する。

- 滑らかさ: 高速な振動やジッタがない。
- 衝突回避: エージェント同士の重なりが多すぎない。
- 壁挙動: 壁に入らず、壁際で振動せず、自然に回避または壁沿いに移動する。
- 流れの連続性: ランダムに散乱せず、妥当な進行方向が保たれている。
- 密度の自然さ: 局所的な密集と分散が実験目的に対して自然である。
- 退出挙動: ROI外へ出たときだけ消失し、有効領域内で突然消えない。

代表的な失敗モードと対処は次の通り。

- 壁付近で振動する: `r_wall` を下げる、`c_wall` を下げる、速度減衰や有効領域への投影を確認する。
- 壁に侵入する: パラメータ調整ではなく、有効領域投影または壁横断判定を修正する。
- 過剰に散乱する: `c_obs` または `r_obs` を下げる。
- 密集・重なりが多い: `c_obs` または `r_obs` を少し上げる。
- 壁を無視しているように見える: `c_wall` または `r_wall` を少し上げる。ただし `dt=1` では原則 `r_wall <= 1.0` にする。

## Codexに計算させる定量指標

軌跡出力がある場合は、Codexに次を計算させる。

- `mean_speed`、`p95_speed`、最大速度。
- 速度がある場合は `mean_acc`、`p95_acc`、最大加速度。
- フレームごとの近接衝突回数。距離 `< 2 * radius` を目安にする。
- 壁違反回数。点が `walls` 内部、または `free_area` 外部に存在する回数。
- ROI退出数と平均滞在時間。
- 経路の迂回度。経路長を始点終点間距離で割る。
- ジッタ指標。連続ステップ間の進行方向変化が大きい回数または平均値。

壁違反回数は強い制約として扱う。継続的な壁侵入がある候補は、ユーザーが意図的に侵入アーティファクトを調べている場合を除き、不合格にする。

## 出力方針

ユーザーへ回答するときは、次を含める。

- そのまま貼り付けられるPythonのパラメータリスト。
- 探索範囲をその値にした理由。
- 予想される失敗モード。
- 必要に応じてCodex向けプロンプト、または実験実行スクリプトの作成提案。
- 次に行う狭い探索範囲の提案。

説明は実用的かつ簡潔にする。ユーザーのPythonプロジェクトに貼り付けやすいコードブロックを優先する。
